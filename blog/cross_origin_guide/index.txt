14:T6d37,<h1>跨域（Cross-Origin）详解与解决方案</h1>
<h2>一、什么是跨域？</h2>
<h3>1.1 同源策略（Same-Origin Policy）</h3>
<p><strong>同源策略</strong>是浏览器最核心的安全策略之一，由 Netscape 公司于 1995 年引入。它限制了一个源（origin）的文档或脚本如何与另一个源的资源进行交互。这是一个用于隔离潜在恶意文件的重要安全机制。</p>
<blockquote>
<p><strong>同源的定义</strong>：如果两个 URL 的**协议（protocol）、域名（host）、端口（port）**都相同，则它们是同源的。</p>
</blockquote>

<th align="left">与 <code>http://www.example.com:80/page.html</code> 是否同源</th>

</thead>

<h3>1.2 跨域的场景</h3>
<p>浏览器在执行以下操作时，会受到同源策略的限制：</p>
<ul>
<li><strong>AJAX/Fetch 请求</strong>：XMLHttpRequest 和 Fetch API 默认只能请求同源资源</li>
<li><strong>Cookie、LocalStorage、IndexedDB</strong>：无法读取非同源的存储数据</li>
<li><strong>DOM 操作</strong>：无法获取非同源页面的 DOM 节点</li>
<li><strong>Canvas/WebGL</strong>：使用跨域图片时可能被污染（tainted canvas）</li>
</ul>
<blockquote>
<p><strong>注意</strong>：跨域限制是<strong>浏览器的行为</strong>，服务器之间或命令行工具（如 curl、Postman）请求不受同源策略限制。</p>
</blockquote>
<hr>
<h2>二、AJAX 跨域解决方法</h2>
<h3>2.1 JSONP（JSON with Padding）</h3>
<p>JSONP 是一种利用 <code>&#x3C;script></code> 标签不受同源策略限制特性的跨域解决方案。</p>
<p><strong>原理</strong>：</p>
<ul>
<li><code>&#x3C;script></code>、<code>&#x3C;img></code>、<code>&#x3C;link></code> 等标签的 <code>src</code> 属性可以加载跨域资源</li>
<li>服务端将数据包装在一个回调函数中返回</li>
<li>浏览器执行该回调函数，获取数据</li>
</ul>
<p><strong>前端代码示例</strong>：</p>
<pre><code class="language-javascript">function handleResponse(data) {
  console.log('获取到的数据:', data);
}

// 动态创建 script 标签
function jsonpRequest(url, callbackName) {
  const script = document.createElement('script');
  script.src = `${url}?callback=${callbackName}`;
  document.body.appendChild(script);
  
  // 清理
  script.onload = function() {
    document.body.removeChild(script);
  };
}

// 使用
jsonpRequest('https://api.example.com/data', 'handleResponse');
</code></pre>
<p><strong>服务端返回格式</strong>：</p>
<pre><code class="language-javascript">handleResponse({ "name": "张三", "age": 25 });
</code></pre>
<p><strong>JSONP 的优缺点</strong>：</p>
<table>
<thead>
<tr>
<th align="left">优点</th>
<th align="left">缺点</th>
</tr>
</thead>
<tbody>
<tr>
<td align="left">兼容性好，支持老旧浏览器</td>
<td align="left">只支持 GET 请求，不支持 POST/PUT/DELETE</td>
</tr>
<tr>
<td align="left">实现简单，无需服务端复杂配置</td>
<td align="left">存在 XSS 安全风险（服务端注入恶意代码）</td>
</tr>
<tr>
<td align="left">不需要浏览器端 CORS 支持</td>
<td align="left">无法获取详细的错误信息</td>
</tr>
<tr>
<td align="left"></td>
<td align="left">需要服务端配合支持</td>
</tr>
</tbody>
</table>
<blockquote>
<p><strong>安全提醒</strong>：JSONP 存在被恶意网站利用的风险，现在已逐渐被 CORS 取代，建议优先使用 CORS。</p>
</blockquote>
<h3>2.2 跨域资源共享（CORS）</h3>
<p>CORS（Cross-Origin Resource Sharing）是 W3C 标准，目前最主流的跨域解决方案。它允许服务器通过设置特定的 HTTP 响应头来告知浏览器是否允许跨域请求。</p>
<h4>CORS 请求分类</h4>
<p><strong>（1）简单请求（Simple Request）</strong></p>
<p>需同时满足以下条件：</p>
<ul>
<li>方法为 <code>GET</code>、<code>HEAD</code>、<code>POST</code> 之一</li>
<li>头部字段仅限于：<code>Accept</code>、<code>Accept-Language</code>、<code>Content-Language</code>、<code>Content-Type</code></li>
<li><code>Content-Type</code> 仅限于：<code>application/x-www-form-urlencoded</code>、<code>multipart/form-data</code>、<code>text/plain</code></li>
</ul>
<p>简单请求流程：</p>
<pre><code>浏览器直接发送请求，自动在请求头中添加 Origin 字段
服务器返回 Access-Control-Allow-Origin 响应头
浏览器判断该响应头是否允许当前源访问
</code></pre>
<p><strong>（2）预检请求（Preflight Request）</strong></p>
<p>对于非简单请求，浏览器会先发送 <code>OPTIONS</code> 预检请求：</p>
<pre><code>OPTIONS /api/data HTTP/1.1
Origin: http://example.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: Content-Type,Authorization
</code></pre>
<p>服务端响应：</p>
<pre><code>HTTP/1.1 204 No Content
Access-Control-Allow-Origin: http://example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400  // 预检结果缓存时间（秒）
</code></pre>
<h4>服务端 CORS 配置示例</h4>
<p><strong>Node.js / Express</strong>：</p>
<pre><code class="language-javascript">const express = require('express');
const app = express();

// 方式一：使用 cors 中间件
const cors = require('cors');
app.use(cors({
  origin: 'http://localhost:3000',  // 允许的源，* 表示允许所有
  methods: ['GET', 'POST', 'PUT', 'DELETE'],  // 允许的方法
  allowedHeaders: ['Content-Type', 'Authorization'],  // 允许的头部
  credentials: true  // 允许携带 Cookie
}));

// 方式二：手动设置响应头
app.use((req, res, next) => {
  res.header('Access-Control-Allow-Origin', 'http://localhost:3000');
  res.header('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE, OPTIONS');
  res.header('Access-Control-Allow-Headers', 'Content-Type, Authorization, X-Requested-With');
  res.header('Access-Control-Allow-Credentials', 'true');
  
  if (req.method === 'OPTIONS') {
    res.sendStatus(200);  // 预检请求直接返回
  } else {
    next();
  }
});
</code></pre>
<p><strong>Nginx 配置</strong>：</p>
<pre><code class="language-nginx">server {
    listen 80;
    server_name api.example.com;
    
    location / {
        # 允许的源
        add_header 'Access-Control-Allow-Origin' 'https://app.example.com' always;
        add_header 'Access-Control-Allow-Methods' 'GET, POST, PUT, DELETE, OPTIONS' always;
        add_header 'Access-Control-Allow-Headers' 'DNT,User-Agent,X-Requested-With,If-Modified-Since,Cache-Control,Content-Type,Range,Authorization' always;
        add_header 'Access-Control-Allow-Credentials' 'true' always;
        add_header 'Access-Control-Max-Age' 1728000 always;
        
        # 处理预检请求
        if ($request_method = 'OPTIONS') {
            return 204;
        }
        
        proxy_pass http://backend_server;
    }
}
</code></pre>
<h4>前端 withCredentials 配置</h4>
<p>当需要携带 Cookie 或认证信息时，前后端都需要配置：</p>
<pre><code class="language-javascript">// XMLHttpRequest
const xhr = new XMLHttpRequest();
xhr.open('GET', 'https://api.example.com/data');
xhr.withCredentials = true;  // 允许携带 Cookie
xhr.send();

// Fetch API
fetch('https://api.example.com/data', {
  method: 'GET',
  credentials: 'include'  // 允许携带 Cookie
})
.then(response => response.json())
.then(data => console.log(data));

// Axios
axios.defaults.withCredentials = true;
// 或单条请求
axios.get('https://api.example.com/data', { withCredentials: true });
</code></pre>
<blockquote>
<p><strong>重要</strong>：当 <code>withCredentials: true</code> 时，服务端<strong>不能</strong>将 <code>Access-Control-Allow-Origin</code> 设置为 <code>*</code>，必须指定具体的域名！</p>
</blockquote>
<h3>2.3 Flash 跨域（已废弃）</h3>
<p>Flash 曾经提供 <code>crossdomain.xml</code> 机制来实现跨域。由于 Flash 已于 2020 年底停止支持，<strong>此方法已不再推荐使用</strong>。</p>
<p>如果维护遗留系统，只需了解 Flash 会读取目标域名根目录下的 <code>crossdomain.xml</code>：</p>
<pre><code class="language-xml">&#x3C;?xml version="1.0"?>
&#x3C;cross-domain-policy>
    &#x3C;allow-access-from domain="*.example.com" />
&#x3C;/cross-domain-policy>
</code></pre>
<h3>2.4 服务端代理（反向代理）</h3>
<p>通过在同源的服务器端设置代理，让服务器代替浏览器访问跨域资源，再将结果返回给浏览器。浏览器始终只与同源的代理服务器通信，完美避开跨域限制。</p>
<p><strong>实现方式</strong>：</p>
<p><strong>（1）开发环境代理（Webpack DevServer）</strong>：</p>
<pre><code class="language-javascript">// vue.config.js 或 webpack.config.js
module.exports = {
  devServer: {
    proxy: {
      '/api': {
        target: 'https://api.example.com',  // 目标 API 服务器
        changeOrigin: true,  // 改变请求源头
        pathRewrite: {
          '^/api': ''  // 重写路径：/api/users → /users
        },
        secure: false,  // 接受 https 自签名证书
        // 可选：添加自定义请求头
        headers: {
          'X-Custom-Header': 'foobar'
        }
      }
    }
  }
};
</code></pre>
<p>前端请求时代码无需任何特殊处理：</p>
<pre><code class="language-javascript">// 直接请求同源的 /api 路径，实际被代理到 https://api.example.com
axios.get('/api/users').then(res => console.log(res));
</code></pre>
<p><strong>（2）生产环境 Nginx 反向代理</strong>：</p>
<pre><code class="language-nginx">server {
    listen 80;
    server_name app.example.com;
    
    # 前端静态资源
    location / {
        root /var/www/app;
        try_files $uri $uri/ /index.html;
    }
    
    # API 代理
    location /api/ {
        proxy_pass https://api.example.com/;
        proxy_set_header Host api.example.com;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
</code></pre>
<p><strong>（3）Node.js 代理服务器</strong>：</p>
<pre><code class="language-javascript">const express = require('express');
const { createProxyMiddleware } = require('http-proxy-middleware');

const app = express();

app.use('/api', createProxyMiddleware({
  target: 'https://api.example.com',
  changeOrigin: true,
  pathRewrite: { '^/api': '' },
  onProxyRes: function(proxyRes, req, res) {
    // 可在这里统一处理响应
  }
}));

app.listen(3000);
</code></pre>
<p><strong>代理方案的优缺点</strong>：</p>

<thead>
<tr>
<th align="left">优点</th>
<th align="left">缺点</th>
</tr>
</thead>
<tbody>
<tr>
<td align="left">无需处理复杂的 CORS 配置</td>
<td align="left">增加了一层网络转发，可能带来延迟</td>
</tr>
<tr>
<td align="left">前端代码无需任何修改</td>
<td align="left">需要额外的服务器资源</td>
</tr>
<tr>
<td align="left">可以统一处理认证、日志、缓存等</td>
<td align="left">需要配置和维护代理服务</td>
</tr>
<tr>
<td align="left">隐藏了真实后端地址，更安全</td>
<td align="left"></td>
</tr>
</tbody>

<h2>三、前端跨域通信（iframe 场景）</h2>
<p>当页面中使用 <code>&#x3C;iframe></code> 嵌入不同源的页面时，两者之间直接的 JavaScript 访问会受到同源策略限制。以下是几种解决方案：</p>
<h3>3.1 document.domain + iframe（仅限子域）</h3>
<p><strong>适用场景</strong>：主域相同，仅子域不同。如 <code>a.example.com</code> 和 <code>b.example.com</code>。</p>
<p><strong>原理</strong>：将两个页面的 <code>document.domain</code> 都设置为主域 <code>example.com</code>，浏览器即认为它们同源。</p>
<p><strong>代码示例</strong>：</p>
<p><code>http://a.example.com/page.html</code>：</p>
<pre><code class="language-html">&#x3C;!DOCTYPE html>
&#x3C;html>
&#x3C;body>
  &#x3C;iframe id="iframe" src="http://b.example.com/page.html">&#x3C;/iframe>
  &#x3C;script>
    // 必须双方都设置
    document.domain = 'example.com';
    
    window.onload = function() {
      const iframe = document.getElementById('iframe');
      // 现在可以访问 iframe 的内容
      console.log(iframe.contentWindow.document.body.innerHTML);
    };
  &#x3C;/script>
&#x3C;/body>
&#x3C;/html>
</code></pre>
<p><code>http://b.example.com/page.html</code>：</p>
<pre><code class="language-html">&#x3C;!DOCTYPE html>
&#x3C;html>
&#x3C;body>
  &#x3C;p>我是 B 页面的内容&#x3C;/p>
  &#x3C;script>
    document.domain = 'example.com';
    // 也可以访问父页面
    console.log(window.parent.document.title);
  &#x3C;/script>
&#x3C;/body>
&#x3C;/html>
</code></pre>
<blockquote>
<p><strong>限制</strong>：</p>
<ul>
<li>只能用于<strong>主域相同、子域不同</strong>的场景</li>
<li>设置 <code>document.domain</code> 后<strong>不可逆</strong>，且只能设置为当前域的父域</li>
<li>现代浏览器中此方法逐渐被淘汰，建议优先使用 <code>postMessage</code></li>
</ul>
</blockquote>
<h3>3.2 location.hash + iframe</h3>
<p><strong>原理</strong>：利用 <code>location.hash</code> 的改变不会触发页面刷新，且可以被同域页面读取的特性。通过中间代理页实现跨域通信。</p>
<p><strong>架构</strong>：</p>
<pre><code>A页面 (a.com) 
  → iframe 嵌套 B页面 (b.com)
    → B页面 iframe 嵌套 C页面 (a.com/proxy.html)  // C与A同源
</code></pre>
<p><strong>代码示例</strong>：</p>
<p><code>http://a.com/a.html</code>（父页面）：</p>
<pre><code class="language-html">&#x3C;iframe id="iframe" src="http://b.com/b.html">&#x3C;/iframe>
&#x3C;script>
  // 1. 向 B 页面传递数据
  const iframe = document.getElementById('iframe');
  
  function sendMessageToB(data) {
    // 通过修改 B 页面的 hash 来传递数据
    iframe.src = `http://b.com/b.html#${encodeURIComponent(JSON.stringify(data))}`;
  }
  
  // 2. 监听 C 代理页面传回的数据
  function checkHash() {
    const hash = location.hash;
    if (hash &#x26;&#x26; hash.length > 1) {
      const data = JSON.parse(decodeURIComponent(hash.substring(1)));
      console.log('收到 B 页面的回复:', data);
      location.hash = '';  // 清理 hash
    }
  }
  
  setInterval(checkHash, 100);  // 轮询监听 hash 变化
  sendMessageToB({ action: 'getUserInfo', userId: 123 });
&#x3C;/script>
</code></pre>
<p><code>http://b.com/b.html</code>（跨域子页面）：</p>
<pre><code class="language-html">&#x3C;script>
  // 1. 监听 hash 变化，获取 A 页面传来的数据
  window.onhashchange = function() {
    const hash = location.hash.substring(1);
    const data = JSON.parse(decodeURIComponent(hash));
    console.log('收到 A 页面的消息:', data);
    
    // 2. 处理数据后，通过同域的 C 页面将结果传回 A
    const result = { status: 'success', userInfo: { name: '张三', age: 25 } };
    
    // C 页面与 A 页面同源（都在 a.com 下）
    const proxyIframe = document.createElement('iframe');
    proxyIframe.src = `http://a.com/proxy.html#${encodeURIComponent(JSON.stringify(result))}`;
    document.body.appendChild(proxyIframe);
  };
&#x3C;/script>
</code></pre>
<p><code>http://a.com/proxy.html</code>（代理页面，与 A 同源）：</p>
<pre><code class="language-html">&#x3C;script>
  // 读取 hash 数据，传递给父页面（A）
  const hash = location.hash;
  if (window.parent !== window) {
    window.parent.parent.location.hash = hash;  // 跨两层 iframe 传给 A
  }
&#x3C;/script>
</code></pre>
<blockquote>
<p><strong>缺点</strong>：</p>
<ul>
<li>实现复杂，需要三个页面配合</li>
<li>使用轮询监听 hash，性能较差</li>
<li>数据暴露在 URL 中，有长度限制且不安全</li>
<li><strong>已过时</strong>，仅作为了解</li>
</ul>
</blockquote>
<h3>3.3 window.name</h3>
<p><strong>原理</strong>：<code>window.name</code> 属性有一个特殊特性——页面跳转后它的值保持不变，且没有同源限制。利用这一特性可以实现跨域数据传递。</p>
<p><strong>代码示例</strong>：</p>
<p><code>http://a.com/a.html</code>（父页面）：</p>
<pre><code class="language-html">&#x3C;iframe id="iframe" src="http://b.com/b.html">&#x3C;/iframe>
&#x3C;script>
  let state = 0;
  const iframe = document.getElementById('iframe');
  
  iframe.onload = function() {
    if (state === 0) {
      // 第一次加载：B 页面正在设置 window.name
      // 将 iframe 指向同源的代理页面，以便读取 name
      iframe.src = 'http://a.com/proxy.html';
      state = 1;
    } else if (state === 1) {
      // 第二次加载：现在可以读取到 window.name 了
      console.log('获取到数据:', iframe.contentWindow.name);
      // 数据使用完后清理
      iframe.contentWindow.name = '';
    }
  };
&#x3C;/script>
</code></pre>
<p><code>http://b.com/b.html</code>（数据页面）：</p>
<pre><code class="language-html">&#x3C;script>
  // 将要传递的数据写入 window.name
  window.name = JSON.stringify({
    users: [
      { id: 1, name: '张三' },
      { id: 2, name: '李四' }
    ],
    total: 100
  });
&#x3C;/script>
</code></pre>
<p><code>http://a.com/proxy.html</code>（空白代理页，仅用于切换同源）：</p>
<pre><code class="language-html">&#x3C;!DOCTYPE html>
&#x3C;html>
&#x3C;body>&#x3C;/body>
&#x3C;/html>
</code></pre>
<blockquote>
<p><strong>缺点</strong>：</p>
<ul>
<li>需要 iframe 两次加载，体验不佳</li>
<li><code>window.name</code> 只能存储字符串，且有大小限制（约 2MB）</li>
<li>安全性较低，其他脚本也能修改 <code>window.name</code></li>
<li><strong>已过时</strong>，建议用 <code>postMessage</code> 替代</li>
</ul>
</blockquote>
<h3>3.4 postMessage（推荐）</h3>
<p><code>postMessage</code> 是 HTML5 引入的现代跨域通信 API，是目前<strong>最推荐</strong>的跨域通信方案。它安全、简洁、功能强大。</p>
<p><strong>API 说明</strong>：</p>
<pre><code class="language-javascript">// 发送消息
otherWindow.postMessage(message, targetOrigin, [transfer]);

/**
 * @param {any} message - 要发送的数据（会被结构化克隆算法序列化）
 * @param {string} targetOrigin - 目标窗口的源，如 'http://example.com'，'*' 表示不限制
 * @param {Transferable[]} [transfer] - 可选，可转移对象
 */

// 接收消息
window.addEventListener('message', function(event) {
  // event.source - 发送消息的窗口引用
  // event.origin - 发送消息的源（协议+域名+端口）
  // event.data - 接收到的数据
});
</code></pre>
<p><strong>代码示例</strong>：</p>
<p><code>http://a.com/parent.html</code>（父页面）：</p>
<pre><code class="language-html">&#x3C;iframe id="iframe" src="http://b.com/child.html">&#x3C;/iframe>
&#x3C;script>
  const iframe = document.getElementById('iframe');
  
  // 等待 iframe 加载完成
  iframe.onload = function() {
    // 向子页面发送消息
    iframe.contentWindow.postMessage(
      { type: 'REQUEST_USER_INFO', userId: 123 },
      'http://b.com'  // 明确指定目标源，不要使用 *
    );
  };
  
  // 监听子页面的回复
  window.addEventListener('message', function(event) {
    // 必须验证消息来源！
    if (event.origin !== 'http://b.com') return;
    
    // 验证消息类型
    if (event.data.type === 'USER_INFO_RESPONSE') {
      console.log('收到用户信息:', event.data.payload);
    }
  });
&#x3C;/script>
</code></pre>
<p><code>http://b.com/child.html</code>（iframe 子页面）：</p>
<pre><code class="language-javascript">// 监听父页面的消息
window.addEventListener('message', function(event) {
  // 安全校验：验证消息来源
  if (event.origin !== 'http://a.com') {
    console.warn('收到来自未知源的消息:', event.origin);
    return;
  }
  
  const { type, userId } = event.data;
  
  if (type === 'REQUEST_USER_INFO') {
    // 处理请求
    const userInfo = getUserInfo(userId);
    
    // 回复父页面
    event.source.postMessage(
      { 
        type: 'USER_INFO_RESPONSE', 
        payload: userInfo 
      },
      event.origin  // 使用 event.origin 确保回复到正确的源
    );
  }
});

function getUserInfo(id) {
  return { id, name: '张三', email: 'zhangsan@example.com' };
}
</code></pre>
<p><strong><code>postMessage</code> 安全最佳实践</strong>：</p>
<pre><code class="language-javascript">window.addEventListener('message', function(event) {
  // 1. 始终验证消息来源
  if (!['https://trusted-parent.com', 'https://partner.com'].includes(event.origin)) {
    return;
  }
  
  // 2. 验证数据格式（防止注入攻击）
  if (!event.data || typeof event.data !== 'object') return;
  
  // 3. 使用白名单校验消息类型
  const ALLOWED_TYPES = ['USER_INFO', 'SETTINGS', 'LOGOUT'];
  if (!ALLOWED_TYPES.includes(event.data.type)) return;
  
  // 4. 处理消息
  handleMessage(event.data, event.source, event.origin);
});
</code></pre>

<h2>四、其他跨域解决方案</h2>
<h3>4.1 WebSocket</h3>
<p>WebSocket 协议本身不受同源策略限制，它的跨域控制通过服务端在握手阶段检查 <code>Origin</code> 头部来实现。</p>
<pre><code class="language-javascript">// 客户端
const ws = new WebSocket('wss://api.example.com/socket');

ws.onopen = () => {
  ws.send(JSON.stringify({ action: 'subscribe', channel: 'notifications' }));
};

ws.onmessage = (event) => {
  console.log('收到消息:', JSON.parse(event.data));
};

// Node.js 服务端（ws 库）
const WebSocket = require('ws');
const wss = new WebSocket.Server({ port: 8080 });

wss.on('connection', (ws, req) => {
  // 验证 Origin
  const origin = req.headers.origin;
  const allowedOrigins = ['https://app.example.com', 'https://admin.example.com'];
  
  if (!allowedOrigins.includes(origin)) {
    ws.close();
    return;
  }
  
  ws.on('message', (message) => {
    console.log('收到:', message.toString());
  });
});
</code></pre>
<h3>4.2 跨域图片与 Canvas</h3>
<p>当 Canvas 使用跨域图片时，需要设置 <code>crossOrigin</code> 属性：</p>
<pre><code class="language-javascript">const img = new Image();
img.crossOrigin = 'anonymous';  // 或使用 'use-credentials'
img.onload = function() {
  const canvas = document.createElement('canvas');
  const ctx = canvas.getContext('2d');
  ctx.drawImage(img, 0, 0);
  
  // 如果未设置 crossOrigin，以下操作会抛出 SECURITY_ERR
  const dataURL = canvas.toDataURL();  // 导出图片
  const pixel = ctx.getImageData(0, 0, 1, 1);  // 读取像素
};
img.src = 'https://cdn.example.com/image.jpg';
</code></pre>
<p>服务端需要返回 CORS 图片头：</p>
<pre><code>Access-Control-Allow-Origin: *
</code></pre>

<h2>六、安全注意事项</h2>
<ol>
<li>
<p><strong>CORS 不是安全机制</strong>：CORS 只是浏览器的访问控制机制，不能替代身份认证和权限校验。服务端仍需验证用户身份。</p>
</li>
<li>
<p>**不要滥用 <code>Access-Control-Allow-Origin: ***</code>：在生产环境中，尽量明确指定允许的域名，尤其是涉及敏感数据或需要携带 Cookie 时。</p>
</li>
<li>
<p><strong><code>postMessage</code> 必须验证 <code>origin</code></strong>：接收方一定要检查 <code>event.origin</code>，避免处理来自恶意网站的消息。</p>
</li>
<li>
<p><strong>避免使用已废弃的方案</strong>：JSONP、<code>location.hash</code>、<code>window.name</code> 等方法存在安全风险或兼容性问题，新项目应避免使用。</p>
</li>
<li>
<p><strong>使用 HTTPS</strong>：生产环境务必启用 HTTPS，防止中间人攻击篡改 CORS 头部。</p>
</li>
</ol>
