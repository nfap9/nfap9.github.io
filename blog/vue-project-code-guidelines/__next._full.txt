<p>这套规范来自一个实际 Vue + TypeScript + Element Plus 项目的迭代总结。它的目标不是写出一份“完美文档”，而是让团队里的每个开发者（包括 AI Agent）在写代码时，能做出一致且可维护的决策。文中既包含原则，也包含可直接执行的行数限制、命名约定、组件选择决策树和审查清单。</p>
</blockquote>
<hr>
<h2>一、核心原则</h2>
<p>在动笔写代码之前，先统一三条最基本的原则。它们会贯穿后续的每一条具体规则。</p>
<h3>1.1 单一职责</h3>
<p>一个组件只做一件事，一个函数只做一件事。当某个文件开始同时处理数据获取、数据转换、UI 渲染和事件分发时，它就具备了被拆分的信号。</p>
<h3>1.2 禁止魔法数字</h3>
<p>所有视觉值必须使用 CSS 变量（全局或组件级），所有逻辑中的硬编码值必须提取为具名常量。魔法数字是技术债中最隐蔽也最难批量替换的一种。</p>
<h3>1.3 优先组合而非重复</h3>
<p>复用项目组件、工具函数和组合式函数。不要复制粘贴相似代码。重复代码会在 AI 或新人参与开发时被再次复制，导致问题呈指数扩散。</p>
<hr>
<h2>二、代码架构规则</h2>
<h3>2.1 组件大小限制</h3>

</tr>

<p>组件太长往往意味着职责过多，阅读、测试和后续修改成本都会上升。</p>
<h3>2.2 组件拆分原则</h3>
<p>满足以下任一条件时，就应当拆分组件：</p>
<ul>
<li><strong>复用</strong>：同一个 UI 模式出现在多个页面或模块中 → 提取到 <code>src/components/</code>。</li>
<li><strong>复杂度</strong>：一个组件同时承担数据获取、转换、渲染、事件处理，或拥有超过 3 个不同视觉区块 → 拆分为子组件。</li>
<li><strong>视图分层</strong>：页面包含多个由标题或分割线分隔的独立业务区块 → 每个区块下沉到页面 <code>components/</code> 目录。</li>
<li><strong>逻辑过重</strong>：一个组件中有超过 100 行纯逻辑（computed、watch、异步函数、数据转换）→ 移到 <code>composables/</code> 或 <code>utils/</code>。</li>
</ul>
<p>推荐的目录结构：</p>
<pre><code>src/
├── components/              → 全局可复用组件
│   ├── BaseDialog.vue
│   ├── StatCard.vue
│   └── ...
├── views/
│   ├── some-page/
│   │   ├── index.vue        → 页面入口（轻量，委托给子组件）
│   │   └── components/      → 页面局部组件
│   │       ├── SearchFilter.vue
│   │       └── DataTable.vue
│   └── ...
</code></pre>
<h3>2.3 函数设计规则</h3>
<ul>
<li><strong>一个函数，一个动作</strong>：要么获取数据、要么转换数据、要么渲染、要么处理事件，不要同时做四件事。</li>
<li><strong>尽早提取</strong>：如果一个函数需要注释来解释其步骤，这些步骤应该成为独立函数。</li>
<li><strong>优先纯函数</strong>：<code>src/utils/</code> 中的工具函数应无副作用。副作用属于事件处理器、生命周期钩子、组合式函数或 Pinia actions。</li>
<li><strong>按意图命名</strong>：<code>fetchUserList</code> 优于 <code>getData</code>，<code>handleSubmit</code> 优于 <code>clickBtn</code>。</li>
</ul>

<h3>3.1 变量类别：只使用这些，禁止硬编码</h3>

<li><strong>背景颜色</strong>：<code>bg</code>、<code>page</code>、<code>overlay</code></li>

<h3>3.6 组件级变量</h3>
<p>全局变量是默认选择。只有在以下场景下，才允许在组件内部定义局部 CSS 变量：</p>
<ul>
<li>组件有独特的内边距 / 尺寸，不适合全局间距层级。</li>
<li>组件有特有的动画时长或缓动曲线。</li>
<li>组件需要局部计算值（如 <code>calc(100% - var(--spacing-lg))</code>）。</li>
<li>复杂组件的子元素需要统一的局部 token。</li>
</ul>
<p>命名规范：</p>
<pre><code>--{component-name}--{property}
--{component-name}__{element}--{property}
</code></pre>
<p>示例：</p>
<pre class="shiki github-dark" style="background-color:#24292e;color:#e1e4e8" tabindex="0"><code><span class="line"><span style="color:#E1E4E8">--base-dialog--header-padding</span></span>
<span class="line"><span style="color:#E1E4E8">--stat-card__value--font-size</span></span></code></pre>
<p>禁止用组件级变量替代语义颜色，或用它逃避全局间距规范。如果一个值在多个组件中重复出现，应提取到全局变量。</p>
<p>决策流程：</p>
<pre><code>需要定义一个值？
  → 全局变量已覆盖？→ 使用全局变量
  → 仅该组件需要？→ 定义组件级变量
  → 多个组件都需要但全局没有？→ 补充到全局变量，然后使用全局变量
</code></pre>

<h2>五、典型布局模式</h2>
<p>一个页面往往不是单一布局，每个由标题或分割线分隔的独立业务区块都是一个独立布局。</p>
<h3>5.1 列表页（表格 + 操作）</h3>
<pre><code>页面头部
  ├─ 标题 + 副标题（左侧）
  └─ 主操作按钮（右侧）
筛选栏
  ├─ 筛选输入框
  ├─ 重置按钮（次要）
  └─ 查询按钮（主要）
数据表格
  ├─ 列（适当处可排序）
  └─ 操作列（小型文字按钮）
分页
  ├─ 信息文本（左侧）
  └─ 页码按钮（右侧）
</code></pre>
<p>Token 要点：页面标题用 <code>font-size-large</code> + <code>font-weight-bold</code> + <code>text-color-primary</code>；主要区块间距用 <code>spacing-2xl</code>；表格使用 <code>el-table</code>（斑马纹已全局，不要加 <code>stripe</code>）。</p>
<h3>5.2 表单页</h3>
<pre><code>页面头部
  ├─ 返回按钮（左侧）
  ├─ 标题
  └─ 副标题（可选）
表单容器
  ├─ 表单字段（标签在输入框上方）
  └─ 提交按钮
      ├─ 取消（次要）
      └─ 提交（主要）
</code></pre>
<p>Token 要点：表单容器限制最大宽度 <code>content-max-width</code>；字段间距用 <code>spacing-xl</code>；提交按钮用 <code>main-button-primary</code> 或 <code>el-button type="primary"</code>。</p>
<h3>5.3 卡片网格 / 仪表盘</h3>
<pre><code>页面头部
  ├─ 标题
  └─ 操作按钮（可选）
卡片网格
  ├─ 卡片 1（.bg-model-card）
  ├─ 卡片 2（.bg-model-card）
  └─ 卡片 N（.bg-model-card）
</code></pre>
<p>响应式策略：小屏单列，中屏 2 列，大屏 3 列或更多。卡片网格间距用 <code>spacing-lg</code> 或 <code>spacing-xl</code>。</p>
<h3>5.4 仪表盘 / 概览（统计 + 图表 + 列表）</h3>
<pre><code>页面标题
统计行
  ├─ 统计卡片 1（StatCard）
  ├─ 统计卡片 2
  └─ ...
内容行
  ├─ 左侧面板（图表或表格）
  └─ 右侧面板（列表或详情）
</code></pre>
<p>统计卡片使用语义颜色（<code>primary</code>、<code>success</code>、<code>danger</code>、<code>info</code>），面板容器使用 <code>.bg-model-card</code>。</p>
<h3>5.5 通用布局规则</h3>
<ol>
<li>主要页面区块间使用 <code>spacing-2xl</code> 或 <code>spacing-3xl</code>。</li>
<li>标准页面内容内边距为 <code>content-padding</code>（24px）或 <code>spacing-2xl</code>。</li>
<li>内容区域遵守 <code>content-max-width</code>（72rem），超宽时居中。</li>
<li>背景层级：页面根节点用 <code>bg-color-page</code>，卡片 / 浮层面板用 <code>bg-color</code>，微妙区块用 <code>fill-color</code> 或 <code>fill-color-light</code>。</li>
</ol>

<h2>七、文件结构提示</h2>
<pre><code>src/styles/variables.less          → 自定义 Design Tokens（无 --el- 前缀）
src/styles/element-plus-theme.less → Element Plus 变量覆盖（桥接层）
src/styles/legacy-variables.less   → 向后兼容别名
src/styles/components.less         → 可复用 CSS 类（.bg-model-card、.main-button）
src/styles/utilities.less          → 工具类（.ellipsis、.flex-center、.text-white）
src/components/                    → 全局可复用组件（BaseDialog、StatCard 等）
src/request/                       → API 请求模块（按业务域）
src/store/modules/                 → Pinia stores（按业务域）
src/composables/                   → 可复用 Vue 组合式逻辑
src/utils/                         → 纯工具函数
src/index.less                     → 全局样式入口
src/main.ts                        → 全局默认 props、应用启动
</code></pre>
<p>桥接层说明：自定义 Token 通过 <code>src/styles/element-plus-theme.less</code> 映射到 Element Plus 变量。项目代码中始终使用不带 <code>--el-</code> 前缀的自定义 Token，让 Element Plus 组件在不需要逐个覆盖的情况下自动匹配品牌规范。</p>
