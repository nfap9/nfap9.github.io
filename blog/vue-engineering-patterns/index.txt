<h2>一、分层架构与依赖治理</h2>
<p><strong>核心思想：用物理边界约束架构，防止代码腐化。</strong></p>
<p>Vue 源码的包结构是工程化设计的典范。编译器（<code>@vue/compiler-core</code>）和运行时（<code>@vue/runtime-core</code>）是两个独立的包，二者互不直接引用，只通过共享的 <code>@vue/shared</code> 进行通信。这种单向依赖图确保了架构的清晰性。</p>
<h3>Vue 的依赖治理实践</h3>
<table>

<tr>
<th>实践</th>
<th>具体做法</th>
<th>可借鉴点</th>

<tr>
<td>单向依赖图</td>
<td>编译器 → 共享包 ← 运行时，二者互不引用</td>
<td>同层级模块禁止互相依赖</td>
</tr>
<tr>
<td>共享包收敛</td>
<td>所有跨包工具收敛到 <code>@vue/shared</code></td>
<td>提取公共依赖，避免重复和循环引用</td>
</tr>

<h3>前端项目三层划分建议</h3>
<p>基于 Vue 的设计思想，前端项目可以按以下三层组织：</p>
<pre><code>┌─────────────────────────────────────┐
│     视图层 (UI Layer)               │  页面组件、通用组件库
├─────────────────────────────────────┤
│     业务层 (Business Layer)          │  状态管理、业务逻辑、hooks
├─────────────────────────────────────┤
│     基础层 (Infrastructure Layer)    │  工具函数、API 封装、类型定义、常量
└─────────────────────────────────────────────────────────────┘
         依赖方向：上层可引用下层，下层禁止引用上层
</code></pre>
<p><strong>依赖规则</strong>：</p>
<ul>
<li>✅ 视图层 → 业务层、基础层</li>
<li>✅ 业务层 → 基础层</li>
<li>❌ 禁止同层级相互引用（如业务模块 A 引用业务模块 B）</li>
<li>❌ 禁止反向依赖（如基础层引用业务层）</li>
</ul>
<h3>用 ESLint 强制约束依赖方向</h3>
<p>安装插件：</p>
<pre><code class="language-bash">pnpm add -D eslint-plugin-import
</code></pre>
<p>配置 <code>.eslintrc.cjs</code>：</p>
<pre><code class="language-js">module.exports = {
  rules: {
    'import/no-restricted-paths': ['error', {
      zones: [
        // 规则1：基础层（utils）禁止引用业务层
        {
          target: 'src/utils',
          from: 'src/business',
          message: '基础设施层禁止依赖业务层',
        },
        // 规则2：基础层禁止引用视图层
        {
          target: 'src/utils',
          from: 'src/views',
          message: '基础设施层禁止依赖视图层',
        },
        // 规则3：业务层禁止引用视图层
        {
          target: 'src/business',
          from: 'src/views',
          message: '业务层禁止依赖视图层',
        },
        // 规则4：同层级业务模块间禁止互相引用
        {
          target: 'src/business/module-a',
          from: 'src/business/module-b',
          message: '同层级业务模块禁止互相依赖，公共逻辑应下沉到 utils',
        },
      ],
    }],
  },
};
</code></pre>
<blockquote>
<p>💡 <strong>如何落地</strong>：先配置规则为 <code>warn</code> 级别，让团队有时间重构现有代码，清理完毕后再切换为 <code>error</code>。</p>
</blockquote>
<h3>Monorepo 依赖治理进阶</h3>
<p>在 Monorepo 场景下，依赖治理需要额外关注三个问题：</p>
<p><strong>1. 共享包版本同步</strong></p>
<pre><code class="language-json">// packages/shared/package.json
{
  "name": "@my-org/shared",
  "version": "1.2.0"
}

// packages/app/package.json
{
  "dependencies": {
    "@my-org/shared": "workspace:^"
  }
}
</code></pre>
<p>使用 pnpm workspace 协议（<code>workspace:^</code>），发布时自动替换为实际版本号，避免手动同步。</p>
<p><strong>2. 循环依赖检测</strong></p>
<pre><code class="language-bash"># 使用 madge 检测循环依赖
npx madge --circular src/
</code></pre>
<p>发现循环依赖后，将公共部分提取到 <code>shared</code> 包中打破循环。</p>
<p><strong>3. 共享包内容取舍原则</strong></p>
<ul>
<li><strong>应该进 shared</strong>：纯工具函数（<code>isObject</code>、<code>capitalize</code>）、类型定义、通用常量</li>
<li><strong>不应该进 shared</strong>：业务逻辑、组件实现、框架特定的封装</li>
<li><strong>判断标准</strong>：如果删除该函数会影响多个包的正常运行，它就应该在 shared 中</li>
</ul>

<h2>二、条件编译与环境标志</h2>
<p><strong>核心思想：在构建时消除不需要的代码，不让调试代码和平台特定逻辑进入生产包。</strong></p>
<p>Vue 大量使用全局标志（<code>__DEV__</code>、<code>__SSR__</code>、<code>__TEST__</code>）控制代码分支。这些标志不是普通的运行时变量，而是在构建阶段通过 esbuild 的 <code>define</code> 功能直接替换为布尔值。条件分支在打包后会被 Tree Shaking 彻底移除。</p>
<h3>Vue 的条件编译实现</h3>
<pre><code class="language-js">// 源码中的写法
if (__DEV__) {
  console.warn('Invalid vnode type');
  checkComponentName(validName);
}

// 生产环境构建后，整块代码被直接移除
// （相当于这一段从未存在过）
</code></pre>
<p>Vue 的 <code>rollup.config.js</code> 中通过 <code>define</code> 配置实现替换：</p>
<pre><code class="language-js">// rollup.config.js
const esbuildPlugin = esbuild({
  define: {
    __DEV__: JSON.stringify(!isProduction),
    __SSR__: JSON.stringify(target === 'node'),
    __TEST__: 'false',
  },
});
</code></pre>
<h3>前端项目实践</h3>
<p><strong>1. Vite 中配置构建时环境变量</strong></p>
<pre><code class="language-ts">// vite.config.ts
import { defineConfig } from 'vite';

export default defineConfig(({ mode }) => ({
  define: {
    // 构建时直接替换，不是运行时 process.env 读取
    __DEV__: JSON.stringify(mode === 'development'),
    __MOCK__: JSON.stringify(mode === 'development' || mode === 'staging'),
  },
}));
</code></pre>
<p>在代码中使用：</p>
<pre><code class="language-ts">// types/global.d.ts 中先声明全局变量
declare const __DEV__: boolean;
declare const __MOCK__: boolean;

// 业务代码中使用
if (__DEV__) {
  console.log('[Debug] Current user:', user);
  setupDevTools();
}

if (__MOCK__) {
  import('./mock').then(({ setupMock }) => setupMock());
}
</code></pre>
<p><strong>2. .env 文件管理环境差异</strong></p>
<pre><code class="language-bash"># .env.development
VITE_API_BASE_URL=http://localhost:3000
VITE_UPLOAD_ENDPOINT=http://localhost:3000/upload

# .env.production
VITE_API_BASE_URL=https://api.example.com
VITE_UPLOAD_ENDPOINT=https://cdn.example.com/upload
</code></pre>
<blockquote>
<p>⚠️ <strong>关键区别</strong>：<code>define</code> 中的变量在构建时被替换（代码会直接消失），而 <code>import.meta.env</code> 中的变量是运行时读取的字符串。调试代码和开发期工具应该用 <code>define</code>，API 地址等运行时配置用 <code>.env</code>。</p>
</blockquote>
<p><strong>3. 验证产物是否干净</strong></p>
<pre><code class="language-bash"># 构建后搜索残留的开发代码
grep -n "console.log\|console.warn" dist/assets/*.js

# 或使用 rollup-plugin-visualizer 分析产物组成
npx vite-bundle-visualizer
</code></pre>
<p>理想的构建产物中，<code>__DEV__</code> 包裹的代码块应该完全消失，不占任何体积。</p>

<h2>三、平台无关的测试策略</h2>
<p><strong>核心思想：测试粒度与运行时环境解耦，能用 Node 跑的测试绝不走浏览器。</strong></p>
<h3>Vue 的三层测试架构</h3>

<p>Vue 最巧妙的设计是 <code>runtime-test</code> 包——用纯 JavaScript 对象模拟 DOM，让 VDOM 的 <code>mount</code>、<code>patch</code> 等核心逻辑在 Node 环境中即可完成测试，完全不依赖 jsdom 或浏览器。</p>
<h3>runtime-test 的核心原理</h3>
<pre><code class="language-ts">// 用普通 JS 对象模拟 DOM 节点
interface TestElement {
  nodeType: number;
  tagName: string;
  props: Record&#x3C;string, any>;
  children: TestNode[];
  parentNode: TestNode | null;
  eventListeners: Record&#x3C;string, Function[]>;
}

function createElement(tag: string): TestElement {
  return {
    nodeType: 1,
    tagName: tag.toUpperCase(),
    props: {},
    children: [],
    parentNode: null,
    eventListeners: {},
  };
}

function patchVNode(oldVNode: TestNode, newVNode: TestNode) {
  // VDOM diff 逻辑：完全在纯 JS 环境中执行
  // 不涉及任何浏览器 API
}
</code></pre>
<p>借助这个模拟层，Vue 的 VDOM 测试可以在 <strong>毫秒级</strong> 完成，而不需要启动 jsdom。</p>
<h3>前端项目实践</h3>
<p><strong>1. Vitest 统一测试框架配置</strong></p>
<pre><code class="language-ts">// vitest.config.ts

      thresholds: {
        lines: 80,
        functions: 80,
        branches: 70,
        statements: 80,
      },
    },
  },
});
</code></pre>

<p><strong>3. CI 中分层执行策略</strong></p>
<pre><code class="language-yaml"># .github/workflows/ci.yml
name: CI

on: [push, pull_request]

jobs:
  # PR 阶段只跑单元测试（最快，2-3 分钟）
  unit-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: pnpm install
      - run: pnpm test:unit        # 只跑 Node 环境的测试

  # 合并到 main 后跑全量测试
  full-test:
    if: github.ref == 'refs/heads/main'
    needs: unit-test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: pnpm install
      - run: pnpm test:unit
      - run: pnpm test:component   # jsdom 测试
      - run: pnpm test:e2e         # Playwright 测试
</code></pre>
<blockquote>
<p>💡 <strong>关键原则</strong>：</p>
<ul>
<li><strong>测试下沉</strong>：能用 Node 就不走 jsdom，能走 jsdom 就不开浏览器</li>
<li><strong>80/20 法则</strong>：80% 的测试应该是单元测试，它们运行快、定位准</li>
<li><strong>给测试加 <code>data-testid</code></strong>：不要用 class 或文本来定位元素，避免样式重构导致测试失败</li>
</ul>
</blockquote>

<h2>四、代码规范的三层防护</h2>
<p><strong>核心思想：规范不是写在文档里的建议，而是焊死在工具链里的强制约束。</strong></p>
<p>Vue 的 ESLint 配置是编译时的硬约束，不是可忽略的建议。这套思路推广到前端项目中，可以构建三层防护体系。</p>
<h3>三层防护链路</h3>
<pre><code>第一层：编辑器实时提示  →  第二层：提交前自动修复  →  第三层：CI 红线拦截
    （最快反馈）              （兜底机制）               （最终防线）
</code></pre>
<h3>第一层：编辑器实时提示</h3>
<p>VSCode 配置 <code>.vscode/settings.json</code>：</p>
<pre><code class="language-json">{
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": "explicit"
  },
  "eslint.validate": ["javascript", "typescript", "vue"]
}
</code></pre>
<h3>第二层：pre-commit Hook</h3>
<p>安装依赖：</p>
<pre><code class="language-bash">pnpm add -D husky lint-staged

# Husky v9 初始化
npx husky init
</code></pre>
<p>配置 <code>package.json</code>：</p>
<pre><code class="language-json">{
  "lint-staged": {
    "*.{ts,vue}": ["eslint --fix"],
    "*.{css,scss}": ["stylelint --fix"],
    "*": ["prettier --write --ignore-unknown"]
  }
}
</code></pre>
<blockquote>
<p>⚠️ <strong>注意</strong>：lint-staged v10+ 会自动处理 <code>git add</code>，不需要在配置中手动添加。</p>
</blockquote>
<p>创建 hook：</p>
<pre><code class="language-bash"># Husky v9 语法
echo 'npx lint-staged' > .husky/pre-commit
</code></pre>
<h3>第三层：CI 强制检查</h3>
<pre><code class="language-yaml"># .github/workflows/lint.yml
name: Code Quality

on: [push, pull_request]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v2

        with:
          cache: 'pnpm'

      - run: pnpm lint
      - run: pnpm type-check    # 同时跑 TypeScript 类型检查

<pre><code class="language-ts">// vite.config.ts

<h2>五、Commit 规范与发布自动化</h2>
<p><strong>核心思想：让提交信息成为可解析的数据源，驱动 CHANGELOG 生成和版本管理。</strong></p>
<h3>Commit 规范格式</h3>
<p>采用 Angular Commit Convention：</p>
<pre><code>type(scope): subject

[可选的 body]

[可选的 footer]
</code></pre>
<p>示例：</p>
<pre><code>feat(compiler): add template optimization for v-if

The compiler now generates optimized code for v-if
blocks with single static children.

Closes #123
</code></pre>
<h3>用 Husky 拦截不规范提交</h3>
<pre><code class="language-js">// scripts/verify-commit.js
const fs = require('fs');
const path = require('path');

const msgPath = path.resolve('.git/COMMIT_EDITMSG');
const msg = fs.readFileSync(msgPath, 'utf-8').trim();

const commitRE = /^(revert: )?(feat|fix|docs|dx|style|refactor|perf|test|workflow|build|ci|chore|types|wip)(\(.+\))?: .{1,50}/;

if (!commitRE.test(msg)) {
  console.error('\n  ❌ Invalid commit message format.\n');
  console.error('  Expected format: type(scope): subject\n');
  console.error('  Valid types:');
  console.error('    feat     → New feature (triggers minor version)');
  console.error('    fix      → Bug fix (triggers patch version)');
  console.error('    refactor → Code change that neither fixes a bug nor adds a feature');
  console.error('    docs     → Documentation changes');
  console.error('    test     → Adding or correcting tests');
  console.error('    chore    → Build tools and dependency changes\n');
  console.error('  Examples:');
  console.error('    feat(compiler): add template optimization');
  console.error('    fix(v-model): handle blur event correctly');
  console.error('    docs: update installation guide\n');
  process.exit(1);
}
</code></pre>
<p>配置 hook：</p>
<pre><code class="language-bash"># Husky v9
echo 'node scripts/verify-commit.js' > .husky/commit-msg
</code></pre>

<h3>自动化发布流水线</h3>
<pre><code class="language-js">// scripts/release.js
const { prompt } = require('enquirer');
const semver = require('semver');
const { execSync } = require('child_process');
const fs = require('fs');
const path = require('path');

async function main() {
  const pkg = JSON.parse(fs.readFileSync('package.json', 'utf-8'));
  const current = pkg.version;
  const increments = ['patch', 'minor', 'major'];

  const { release } = await prompt({
    type: 'select',
    name: 'release',
    message: `Current: v${current}. Select release type:`,
    choices: increments.map(i => ({
      name: `${i} (${semver.inc(current, i)})`,
      value: i,
    })),
  });

  const type = release.match(/^(\w+)/)[1];
  const target = semver.inc(current, type);

  const { confirm } = await prompt({
    type: 'confirm',
    name: 'confirm',
    message: `Release v${target}?`,
    initial: true,
  });

  if (!confirm) {
    console.log('Cancelled.');
    process.exit(0);
  }

  // 更新版本号
  pkg.version = target;
  fs.writeFileSync('package.json', JSON.stringify(pkg, null, 2) + '\n');

  // 生成 CHANGELOG
  execSync('pnpm changelog', { stdio: 'inherit' });

  // Git 提交、打标签、推送
  execSync(`git add -A &#x26;&#x26; git commit -m "release: v${target}"`, { stdio: 'inherit' });
  execSync(`git tag v${target}`);
  execSync('git push &#x26;&#x26; git push --tags', { stdio: 'inherit' });

  console.log(`\n🎉 v${target} released!`);
}

main().catch(err => {
  console.error(err);
  process.exit(1);
});
</code></pre>
<p><code>package.json</code> 配置：</p>
<pre><code class="language-json">{
  "scripts": {
    "release": "node scripts/release.js",
    "changelog": "conventional-changelog -p angular -i CHANGELOG.md -s"
  }
}
</code></pre>

<h3>1. 画好"依赖地图"</h3>
<p>用 <code>import/no-restricted-paths</code> 明确每个目录的职责和依赖方向。这是防止代码腐化最有效、成本最低的手段。</p>
<h3>2. 测试分层守护</h3>
<p>单元测试（Node）→ 组件测试（jsdom）→ E2E（浏览器），按速度由快到慢、覆盖范围由小到大递进。CI 中按分支策略分层执行，PR 阶段只跑单元测试，合并后跑全量。</p>
<h3>3. 规范焊死在工具链</h3>
<p>能自动化的不要靠人记，能编译时拦截的不要留到运行时。Commit 规范 + CHANGELOG 自动生成 + 语义化版本，让发布变成一条命令的事。</p>
